Introduction & Objective

The Concept

Traditional trash cans are passive objects. We asked ourselves: "Why should you go to the trash can? Why can't the trash can actively come to catch the trash?"

Driven by this idea, we designed an active robotic platform. Our core objective was to validate the feasibility of Visual Servoing on a trash can robot‚Äîmaking it see the target and move in real-time to intercept it.

üèÜ Key Success Criteria

  • Omni-Motion: Instant lateral movement (strafing) without turning.
  • Real-time AI: High-frame-rate detection (~18 FPS) on Raspberry Pi.
  • Closed-Loop Control: A fast "SENSE -> DECIDE -> ACT" loop (~20Hz).

Hardware Evolution

Phase 1: The Two-Wheel Limitation

Initially, we prototyped with a standard 2-Wheel Differential Drive to validate the circuit design. However, testing revealed a fatal flaw: if a target fell to the side, the robot had to turn first, then move forward.

This latency was unacceptable for an interception task‚Äîit simply couldn't catch anything in time.

Phase 2: Three-Wheel Omni Upgrade

To achieve true agility, we completely rebuilt the chassis using a 120¬∞ Three-Wheel Omni-Directional Layout. By arranging motors at 120-degree intervals, the robot can move along any vector \( (V_x, V_y) \) instantly, without changing its heading.

Precision & Integration

We designed custom 3D-printed structural components to ensure mechanical precision.

The Base Plate (top image) serves as the main chassis deck, providing a rigid platform.

The Custom Motor Mount (bottom image) is critical for performance. It locks motors at exactly 120¬∞ angles to ensure kinematic accuracy and reduce vibration.

Software & Algorithm Design

To achieve real-time response on limited hardware, we architected a high-performance C++ system following a strict SENSE → DECIDE → ACT loop running at ~20Hz.

1. SENSE: Edge AI & Multithreading Architecture

Running deep learning on an edge device like the Raspberry Pi 4B is challenging. To achieve real-time performance, we implemented three layers of optimization:

  • The Model: Custom YOLO-FastestV2 trained on Google Colab (Tesla T4) specifically for hands.
  • Inference Acceleration: NCNN framework with OpenMP 4-thread parallel acceleration.
  • Hardware Overclocking: CPU boosted from 1.5GHz to 2.0GHz with active cooling.

Pipeline Optimization (Code)

We decoupled the system into parallel threads to prevent the heavy detection loop from blocking motor control.

# /boot/config.txt
over_voltage=6
arm_freq=2000  # Overclock to 2.0GHz

// src/main.cpp - Parallel Threads
std::thread server_thread(http_server);
yoloF2.init(false); 
net.opt.num_threads = 4; // NCNN OpenMP

2. DECIDE: Visual Servoing (PID)

We use Image-Based Visual Servoing (IBVS). The logic isn't just "move to center," but involves coordinate transformation and dynamic speed adjustment.

Coordinate & "Rush" Logic

The camera faces upward, requiring coordinate inversion. We also tuned the PID with an aggressive P-term for a "Rush Effect".

// src/vision/visual_servo.cpp
float error_x = target_x - smooth_x;
float error_y = smooth_y - target_y;

// Aggressive P-control for "Rush" effect
float speed_multiplier = 4.0f; 
vx *= speed_multiplier;
vy *= speed_multiplier;

Control Strategy

  • Input: Pixel error vector \((E_x, E_y)\).
  • Output: Velocity vector \((V_x, V_y)\).
  • Stability: Adaptive Dead Zone (15-35px) prevents motor jitter when the target is locked.

3. ACT: The "Engineering Hack"

üö® The Challenge: We combined Mecanum wheels (45¬∞ rollers) with a 120¬∞ Omni-layout.
Physics dictated that lateral forces \( F_x \) would generate an unwanted torque \( \tau \), causing the robot to spin uncontrollably while trying to move straight.

üí° The Solution: Instead of rebuilding the chassis, we derived a software compensation matrix. We injected a 1.5x gain and inverted the vector to forcefully cancel the rotation.

This "Software Trim" effectively patched the hardware limitation, allowing perfectly straight movement.

// src/control/kinematics.cpp
float v1 = cos(240¬∞)*vx + sin(240¬∞)*vy;
float v2 = cos(120¬∞)*vx + sin(120¬∞)*vy;
float v3 = cos(0¬∞)*vx   + sin(0¬∞)*vy;

// Software Trim to fix Physical Drift
speeds.m1 =  v1 / max_spd;
speeds.m2 = -v2 / max_spd; 
speeds.m3 = -v3 * 1.5f / max_spd; // The Hack

4. Headless Web Dashboard

Debugging a moving robot requires wireless monitoring. We built a lightweight C++ HTTP Server that streams MJPEG video and system stats (CPU/Temp) to any browser.

  • Zero-Copy Streaming: Direct memory buffer to socket.
  • Low Latency: <50ms lag (vs 200ms+ with Python Flask).
  • Safety: Real-time thermal monitoring for overclocked CPU.
